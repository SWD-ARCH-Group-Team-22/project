### Knowledge dependencies from co-change analysis

As a first step of the dependency analysis, we focused on the knowledge dependencies of Freeplane. Since the project is large, we decided not to start immediately from the source code. Reading the code without any initial direction would have been much longer and less effective. Instead, we first used the Git history to understand which parts of the system usually changed together.

To do this, we automatically generated co-change reports from the commit history. These reports show which pairs of files were modified in the same commits, and how many times this happened. If two files often appear together in commits, it does not automatically mean that one imports the other or that there is a direct code dependency. However, it is a useful signal: probably, when developers changed one of them, they also had to understand or update the other. For this reason, we used co-change as an indication of possible knowledge dependencies.

This idea is connected to what we discussed in class about software complexity. The professor explained that dependencies are not always wrong, but they become dangerous when they increase the cognitive load, because developers need to know too many parts of the system, or when they cause change amplification, because a small change spreads to many files. In our case, the co-change analysis was used as a first map of the system: it helped us choose the areas that deserve a deeper inspection in the code.

The reports were generated for three time windows: the last 5 years, the last 10 years and the full repository history. We used them to distinguish recent co-change from relations that have been stable for a longer time. The full-history report was useful as background, but we read it more carefully because it also includes older paths and older project structures.

Instead of reporting all the pairs one by one, we grouped them into functional areas. This choice follows the same reasoning behind the Common Closure Principle: if classes tend to change together, they may belong to the same responsibility or at least to the same maintenance concern. This does not mean that every co-change is a problem. Sometimes it simply shows good cohesion: files change together because they are part of the same feature. In other cases, especially when the files are in different modules, co-change may suggest a stronger hidden dependency.

The clearest cluster is the Swing map view. Classes such as `MapView`, `NodeView`, `MainView`, `NodeViewFactory` and the layout classes appear together in all three reports. This is not surprising: Freeplane is mainly a mind-mapping application, so the way maps and nodes are shown on screen is a central part of the system. Our first interpretation is that this is probably normal cohesion inside the main UI subsystem. Still, because this area is so central, we will later check in the code whether these classes are well grouped or whether the map view depends on too many other parts.

Another important cluster is the outline subsystem, with classes such as `ScrollableTreePanel`, `BreadcrumbPanel`, `OutlinePane`, `MapAwareOutlinePane` and `OutlineController`. This area is interesting because the outline is another way of looking at the same map content, but in a more tree-like structure. For this reason, it may naturally need to stay aligned with the main map view: if selection, scrolling or node positioning changes in the map, the outline may also need updates. The code analysis will help us understand whether this is a clean feature dependency or whether the outline is too tightly connected to the map view.

The most interesting case is the relation between the public API and the scripting plugin. We can see this because the repository has a separate `freeplane_api` module and also a separate `freeplane_plugin_script` module, but classes such as `Node`, `NodeRO` and `NodeProxy` still appear together in the co-change reports. In simple terms, scripts need a way to access Freeplane concepts such as nodes and maps. If the public node API changes, the proxy used by scripts may also need to change. This could be a good and controlled dependency if the scripting plugin uses only public interfaces. It would be more problematic if the plugin also depends on internal implementation details.

Finally, the text rendering plugins show another useful case. `FormulaTextTransformer`, `LatexRenderer` and `MarkdownRenderer` belong to different plugin areas, but they all deal with the same general concern: transforming or rendering text inside nodes. So their co-change may mean that, when Freeplane changes how node text is handled, more than one text-related plugin has to be updated.

This first analysis does not prove design problems by itself. It gives us a set of meaningful areas to inspect next in the code dependencies analysis.
