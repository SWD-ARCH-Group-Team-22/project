
# How to Build and Run Freeplane

This guide walks you through the process of cloning the repository, compiling the code with Gradle, launching the application, and generating a distribution package.

##  Prerequisites

Before you begin, ensure your system meets the following requirements:
* **Java:** Version 11 or later.
* **Gradle:** Version 6.0 or higher.

---

## Step-by-Step Instructions

### 1. Download and Setup Gradle
1. Download a compatible version of Gradle from the [official releases page](https://gradle.org/releases/).
2. Unzip the downloaded archive into your home directory (or another preferred location). 

### 2. Clone the Repository
Open your terminal, clone the official Freeplane source code, and navigate into the directory:

```bash
git clone [https://github.com/freeplane/freeplane.git](https://github.com/freeplane/freeplane.git)
cd freeplane

```

### 3. Build the Project

_Note: Gradle wrapper files (`gradlew`) are intentionally ignored in the repository's version control. You must use your locally installed Gradle executable._

Execute the `build` command, replacing the path below with the actual location where you unzipped Gradle:

Bash

```
~/Downloads/gradle-6.0.1/bin/gradle build

```

### 4. Run the Application

Once the build process completes successfully, navigate to the binary output folder and run the executable specific to your operating system:

-   **For Linux / macOS:**
    
    Bash
    
    ```
    cd BIN
    ./freeplane.sh
    
    ```
    
-   **For Windows:**
    
    DOS
    
    ```
    cd BIN
    freeplane.bat
    
    ```
    



## Create Package

If you want to create a clean, standalone distribution package of Freeplane, you can chain the `clean`, `build`, and `dist` tasks together:

Bash

```
~/Downloads/gradle-6.0.1/bin/gradle clean build dist

```

After the command finishes, your generated packages will be located in the `DIST` directory. You can view them by running:

Bash

```
ls DIST

```

## Setting Up VS Code for Freeplane Development

While the [official documentation](https://docs.freeplane.org/coding/IDE_setup.html) covers Eclipse, IntelliJ, and Netbeans, you can follow these steps to configure VS Code for a smooth development experience.

## 1. Prerequisites

Before starting, ensure you have the following installed:

-   **JDK** (Check the `build.gradle` file for the required version—usually Java 11 or 17).
    
-   **Visual Studio Code**.
    
-   **Extension Pack for Java** (included in the Extension Pack).
    
    

## 2. Import the Project

1.  Open VS Code and select **File > Open Folder...**.
    
2.  Navigate to your local Freeplane repository (the root folder containing `settings.gradle`) and open it.
    
3.  Wait for the **Java Language Server** to activate. You will see a progress bar in the bottom status bar. VS Code will automatically detect the Gradle project structure.
    

## 3. Initial Build (Crucial Step)

Freeplane uses generated code (like lexers). If you don't run a build first, VS Code will show many "symbol not found" errors.

1.  Open the integrated terminal (`Ctrl` + `` ` ``).
    
2.  Run the following command:
    
    Bash
    
    ```
    ./gradlew clean build -x test
    
    ```
    
    _Note: If you have old `MANIFEST.MF` files from previous builds, delete them as suggested in the official docs._
    

## 4. Configuring the Launch (Debug)

To run and debug Freeplane directly from VS Code, you need a `launch.json` file.

1.  Go to the **Run and Debug** view (`Ctrl+Shift+D`).
    
2.  Click on **create a launch.json file** and select **Java**.
    
3.  Paste the following configuration (adjusting the `projectName` if necessary):
    

JSON

```
{
    "version": "0.2.0",
    "configurations": [
        {
            "type": "java",
            "name": "Debug Freeplane",
            "request": "launch",
            "mainClass": "org.freeplane.launcher.Launcher",
            "projectName": "freeplane_debughelper",
            "vmArgs": "-Dorg.freeplane.basedir=${workspaceFolder}",
            "args": ""
        }
    ]
}

```

## 5. Working with Gradle Tasks

You can use the **Gradle side-bar** (the elephant icon) to access all Freeplane tasks:

-   **Build**: Use the `build` task to compile everything.
    
-   **Run**: Use the `run` task under the `freeplane_debughelper` module to launch the app without the debugger.

### IDE Setup: Eclipse, IntelliJ, and Netbeans

As mentioned, while VS Code is an option, Freeplane provides extensive support for Eclipse, IntelliJ, and Netbeans. For any IDE, if you have executed Gradle builds for previous Freeplane versions, you should manually delete all old `MANIFEST.MF` files generated by build scripts before starting.

#### Eclipse
1. **Project Generation**: Instead of importing Gradle files directly, you can generate all necessary Eclipse project files by running a clean build from your terminal: `gradle clean build cleanEclipse eclipse`.
2. **Importing the Project**: Open your Eclipse IDE, point it to the base project folder (where you executed the Gradle command), and import ALL Freeplane project options from the list. *Note: Please drop the `freeplane_mac` project if you are not working on a macOS environment*.
3. **Running and Debugging**: To launch Freeplane for the first time, locate the launch configuration file within the source tree at `freeplane_debughelper/eclipse/freeplane.launch`. Once launched, you will find the appropriate menu items in the Run and Debug icon menus for future executions.

#### IntelliJ IDEA
1. **Importing the Project**: Open IntelliJ and click **Open**. Point to the `build.gradle` file in the Freeplane checkout directory, click OK, and choose "Open as Project". *(If you are using IntelliJ 2020 or older, point to `settings.gradle` instead)*.
2. **Run Configurations**: IntelliJ requires manual run configurations. Copy the entire `freeplane_debughelper/idea/runConfigurations` directory into the `.idea` folder created by IntelliJ in the project root, then restart the IDE.
3. **Compiler Settings Troubleshooting**: If you get a protected package error regarding `sun.awtshell`, navigate to File -> Settings -> Build -> Compiler -> Java Compiler and uncheck the "use Release" option.
4. **Initial Build**: Execute a top-level Gradle build from the command line first (this generates necessary files like lexers). Afterward, you can launch the app by going to *Run -> Run... -> freeplane*. Ensure the "Use classpath for module" field is set to `freeplane_root.freeplane_debughelper.main`.

#### Netbeans
1. **Importing and Running**: Netbeans can directly import Gradle files using its native Gradle support plug-in. The specific Gradle tasks required for running and debugging from within the IDE are already defined in `freeplane_debughelper/build.gradle`.

### Git Workflow and Version Control

Freeplane uses a distributed revision control system (Git). The global main branch containing all Freeplane sources is `master` (for the Docear branch, it is `docear/trunk`). 

#### Branching Strategy
* **Naming Conventions**: Always create a feature branch for your development tasks. New branches should be named `<username>/<feature>` or simply `<feature>` if multiple developers are collaborating on it. Docear specific branches must be prefixed with `docear/`.
* **Local vs. Remote**: Branches are cheap in Git, so use them for most tasks—especially to request a code review before integrating your work into the `master` branch.

#### Standard Development Cycle
1. **Checkout the Parent Branch**: Switch to the branch you want to base your work on (usually `master`) and ensure it's up to date by pulling the latest changes.
2. **Create a Local Branch**: Run `git checkout -b <username>/<feature>` to create and immediately switch to your new branch.
3. **Develop Locally**: Make your modifications, add them to the staging area (`git add <file>`), and commit them locally (`git commit -m "Your message"`).
4. **Push for the First Time**: Publish your branch to the remote repository so others can see it. Use the command line for this first push to set up tracking properly: `git push -u origin <branchname>`. **Always run push commands with `--dry-run` first to avoid mistakes!**.
5. **Resolve Conflicts**: Periodically merge changes from `master` into your feature branch to keep it up to date and make the final integration easier.
6. **Merge into Master**: Once you finish development and complete a code review, your branch must be merged into `master`. **Important:** If you are a new developer, you must let a more experienced developer perform this final merge.



### Plugin Development

With the transition to the Gradle build system, the old Ant tasks are deprecated, and creating plugins manually has become a straightforward process. 

#### Steps to Create a New Plugin
1. **Naming:** Choose a short name for your plugin. It must start with the mandatory prefix `freeplane_plugin_` (e.g., `freeplane_plugin_foo`).
2. **Configuration:** Include your new plugin in the `settings.gradle` file.
3. **Build File:** Create a `build.gradle` file inside your plugin directory. The easiest way is to copy an existing one from simpler plugins like `freeplane_plugin_latex` or `freeplane_plugin_openmaps`.
4. **Dependencies:** Add your third-party dependencies and OSGi imports (packages needed from core or other plugins) to your `build.gradle` file. If a third-party dependency is not available on maven.org, use `files()` and commit the library to version control inside `freeplane_plugin_foo/lib`.
5. **Activator:** Create your `Activator.java` file (e.g., under `freeplane_plugin_foo/src/main/java/org/freeplane/plugin/foo/Activator.java`). Again, copying an existing one and stripping out what you don't need is the recommended approach.
6. **Version Control Warning:** When committing your work, make sure **not** to commit `.project` or `.classpath` files, as these are auto-generated by Gradle and should remain local.

#### Adding Plugin Options and Actions
* **Option Panel:** Plugins can register their own options by defining a `preferences.xml` file. By convention, this is located at `freeplane_plugin_<name>/org/freeplane/plugin/<name>/preferences.xml`. Remember to also add the corresponding option names to your `Resource_en.properties` file.
* **Actions and Listeners:** For practical examples on how to register actions or icon mouse click listeners, you should study the `OpenMapsRegistration.java` file found within the `freeplane_plugin_openmaps` plugin source code.

### Export using XSLT

Freeplane allows you to apply XSL Transformations (XSLT) to a mindmap to generate a new file, a feature accessible via **File -> Export...**. 

#### Creating Custom XSLT Transformations
You can build your own XSL scripts to implement custom transformations, such as generating custom XHTML and attributing HTML elements with class attributes for CSS styling.
* **File Location:** XSL files must be stored in an `xslt` subfolder. This can be located either in the Freeplane system directory (e.g., under `resources/xslt`) or in the specific user directory (e.g., `~/.freeplane/xslt/`).
* **File Requirements:** The files must use the `.xsl` extension. Crucially, your file must contain the following specific string pattern within its first 5 lines: `MINDMAPEXPORTFILTER extensions description`. 
    * *Example:* `MINDMAPEXPORTFILTER htm;html Meeting Minutes in HTML`. 
    * *Note:* Only the first unique combination of extensions and descriptions will be kept, allowing you to overwrite existing default XSLT sheets with your own custom versions.

#### Troubleshooting: Viewing the XML Input
When designing your transformation, be aware that the XML input fed into the XSLT engine is **different** from the raw content of a standard Freeplane `.mm` file. To view the exact XML structure that your XSL script will process, use the special export sheet named **"Export raw xml with node formatting"**.
